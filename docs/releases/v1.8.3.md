# nexacro-claude-skills v1.8.3

**Release date:** 2026-04-24
**Tag:** `v1.8.3`
**Highlights:** Core modules resolve from tobesoft Nexus, not vendored. v1.8.2's local-snapshot + systemPath approach is superseded.

---

## Why this supersedes v1.8.2

v1.8.2 shipped a partial in-tree mirror of the nexacro first-party modules:

- `core/libs/jakarta/` — 2 jakarta SNAPSHOT jars (520 KB + 202 KB) pinned to timestamp `20260312`
- `core/uiadapter-javax/` — 102 Java files across 3 Spring 5 Maven submodules
- Runners wired via `<scope>system</scope> <systemPath>.../core/libs/...</systemPath> <optional>true</optional>`

Two problems surfaced immediately:

1. **Staleness** — a SNAPSHOT jar frozen at one timestamp diverges from the canonical Nexus artefact, which rolls forward daily. Consumers who pulled v1.8.2 were shipping already-outdated bits.
2. **Asymmetry** — only the jakarta lane had jars; javax xapi/xeni and jakarta uiadapter sources were never in this workspace. The gap was documented but not closable from inside v1.8.2's model.

User direction (paraphrased): *"The plugin should reference the Maven repository and use the latest version of each artifactId."*

v1.8.3 replaces the vendored snapshot approach with Nexus-resolved dependencies.

## What shipped

### 1. Parent pom centralization (`nexacroN-fullstack/pom.xml`)

- **`<repositories>`** block added:
  ```xml
  <repository>
    <id>tobesoft-snapshots</id>
    <url>https://mangosteen.tobesoft.co.kr/nexus/repository/tobesoft-snapshots/</url>
    <releases><enabled>true</enabled></releases>
    <snapshots><enabled>true</enabled><updatePolicy>daily</updatePolicy></snapshots>
  </repository>
  ```
- **10 version properties** — one per nexacro artefact, grouped by lane:
  - jakarta: `nexacro.xapi.jakarta.version`, `nexacro.xeni.jakarta.version`, `nexacro.uiadapter.jakarta.{core,dataaccess,excel}.version`
  - javax: `nexacro.xapi.javax.version`, `nexacro.xeni.javax.version`, `nexacro.uiadapter.spring.{core,dataaccess,excel}.version`
- **`<dependencyManagement>`** — 10 entries (5 jakarta + 5 javax) pin versions to those properties so runners only need `groupId` + `artifactId`.

Bumping a nexacro module is now a one-property edit in one pom.

### 2. Runner poms — bare deps only

**`samples/runners/boot-jdk17-jakarta/pom.xml`:**
```xml
<dependency><groupId>com.nexacro</groupId><artifactId>nexacroN-xapi-jakarta</artifactId></dependency>
<dependency><groupId>com.nexacro</groupId><artifactId>nexacroN-xeni-jakarta</artifactId></dependency>
<dependency><groupId>com.nexacro</groupId><artifactId>uiadapter-jakarta-core</artifactId></dependency>
<dependency><groupId>com.nexacro</groupId><artifactId>uiadapter-jakarta-dataaccess</artifactId></dependency>
<dependency><groupId>com.nexacro</groupId><artifactId>uiadapter-jakarta-excel</artifactId></dependency>
```

**`samples/runners/boot-jdk8-javax/pom.xml`:**
```xml
<dependency><groupId>com.nexacro</groupId><artifactId>nexacroN-xapi</artifactId></dependency>
<dependency><groupId>com.nexacro</groupId><artifactId>nexacroN-xeni</artifactId></dependency>
<dependency><groupId>com.nexacro</groupId><artifactId>uiadapter-spring-core</artifactId></dependency>
<dependency><groupId>com.nexacro</groupId><artifactId>uiadapter-spring-dataaccess</artifactId></dependency>
<dependency><groupId>com.nexacro</groupId><artifactId>uiadapter-spring-excel</artifactId></dependency>
```

No versions. No systemPath. No optional flag. Versions are inherited from the parent's `dependencyManagement`.

**Javax artifactId discovery:** The javax-lane artifactIds do not follow the jakarta naming pattern. Confirmed via Nexus `maven-metadata.xml` probe:

| Jakarta | Javax |
|---|---|
| `nexacroN-xapi-jakarta` | `nexacroN-xapi` |
| `nexacroN-xeni-jakarta` | `nexacroN-xeni` |
| `uiadapter-jakarta-core` | `uiadapter-spring-core` |
| `uiadapter-jakarta-dataaccess` | `uiadapter-spring-dataaccess` |
| `uiadapter-jakarta-excel` | `uiadapter-spring-excel` |

### 3. Local snapshot removed

- **Deleted:** `core/libs/jakarta/` (2 jars)
- **Deleted:** `core/uiadapter-javax/` (102 java files, 3 Maven submodules)
- **Rewritten:** `core/README.md` — 112-line pointer doc covering:
  - why the directory still exists (sparse-checkout path anchor for the starter plugin)
  - where the real modules live (Nexus URL pattern)
  - the 10 artifactIds + version pins (jakarta + javax tables)
  - required `~/.m2/settings.xml` `<server id="tobesoft-snapshots">` credentials
  - how to probe Nexus for the latest version
  - build-verification status

Monorepo tag: **`v0.6.0-core-from-nexus`**.

## Consumer setup

After `/plugin marketplace update nexacro-claude-skills` and `/nexacro-fullstack-starter`, just run `mvn compile` — all 10 nexacro modules resolve from the tobesoft Nexus automatically with no credential setup. No more SNAPSHOT jar staleness, no more systemPath gymnastics.

### Bumping a module version

Probe Nexus for the latest version:

```bash
curl -s https://mangosteen.tobesoft.co.kr/nexus/repository/tobesoft-snapshots/com/nexacro/<artifactId>/maven-metadata.xml \
  | grep -oE '<version>[^<]+' | sort -V | tail -3
```

> **Heads-up:** the `<latest>` tag in `maven-metadata.xml` is not reliable (older re-publishes can reset it). Always sort the full `<version>` list by semver.

Edit the corresponding `<nexacro.*.version>` property in `nexacroN-fullstack/pom.xml`; runners pick it up via inherited `dependencyManagement`.

## Migration from v1.8.2

v1.8.2 consumers who re-scaffold with v1.8.3 get the Nexus-resolved poms automatically. No manual edits required.

If you were already editing the v1.8.2-generated poms to flip `<scope>system</scope>` → `<scope>compile</scope>`: stop. Re-scaffold with v1.8.3 and delete the `core/libs/` and `core/uiadapter-javax/` directories from your copy — they are no longer the source of truth.
